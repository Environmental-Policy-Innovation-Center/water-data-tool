# Tile Cache

Single source of truth for how map tiles are cached, refreshed, busted, and warmed — what each
mechanism does, when it fires automatically, and when it requires a manual workflow run.

- Map layers, styles, and interactions → [MAPPING.md](MAPPING.md)
- ETL import flow → [ETL.md](ETL.md)
- How to run the manual bust/warm workflows → [RUNBOOK.md](RUNBOOK.md#tile-cache) / [DEPLOYMENTS.md](DEPLOYMENTS.md#bust-tile-cache)

---

## Why we cache tiles

The map is drawn from vector tiles (MVT, via PostGIS `ST_AsMVT`) generated by spatial SQL in
`TileGenerator`. Generating a tile on every request would be slow, especially at low zoom where a
single tile can cover thousands of overlapping polygons. Instead, `TilesController#show` checks
the `tile_cache` table first and only falls back to live generation on a miss — the cache **builds
up as the app is used**, and any later request for an already-cached `(layer, z, x, y)` is served
as a fast DB lookup.

The cache is **global and shared across every visitor** — the first user to request a given tile
pays the generation cost; everyone after gets the cached binary instantly, regardless of who
originally generated it.

---

## Cache lifecycle

1. **Cold** — tile has never been requested (or was never cached); PostGIS generates it from
   `service_area_geometries` (slow, especially at low zoom).
2. **Warm** — tile exists in `tile_cache`; returned as a simple DB lookup (fast).
3. **Refreshing** — a data change identifies affected coordinates and overwrites just those rows
   in the background; old cached rows remain readable until replacements are ready.

`TileGenerator.build_tile(z, x, y)` checks all 4 layers (`pws`, `places`, `counties`, `states`) at
once for a given coordinate. Layers already cached are returned immediately; missing layers are
generated, persisted, and returned — a partially-cached coordinate only regenerates what's missing.

`pws` has a second cache key, `pws_low_poly_v1`, used only below `TileGenerator::PWS_MIN_ZOOM`
(z<5). At that zoom the `pws` layer's SQL selects a reduced attribute set (`pwsid`, `stusps` only,
plus simplified geometry) — busting or warming `pws` alone does **not** touch this row; it must be
targeted explicitly when clearing, but warming `pws` at low zoom populates it automatically (see
below).

## Automatic invalidation — two different mechanisms, for two different scopes

### Selective refresh (data changes, narrow scope)

`TileImpact.for_pwsids(pwsids, layers:)` (`app/services/tile_impact.rb`) computes a bounding box
for *only the given pwsids'* geometries, converts it into z0–z8 tile coordinates with a 1-tile edge
margin, and enqueues `TileCacheRefreshJob.perform_later(layer:, z:, coords:)` in batches of 50.
Each job calls `TileGenerator.generate_tile!` for just those coordinates — the refresh *is* the
warm; no separate warming step follows it. Existing cached rows remain readable until their
replacements are written.

This is genuinely surgical: only the tiles containing the changed systems (plus a one-tile margin
for edge cases) are touched — not the whole `pws` layer, not the whole country.

**This only fires when an importer reports what changed.** `Etl::ImportResult#changed_pwsids`
is populated by the importer itself, not derived automatically from the import. `Etl::Importers::EpaSabs`
does this (`changed_pwsids_for`): before upserting, it loads the *existing* DB rows for the incoming
pwsids and compares each of `MAP_FIELDS` field-by-field against the freshly parsed row, collecting
any pwsid that's new or differs. That's the whole mechanism — no CDC, no file diffing, just "load
current values, compare against the new row, right before overwriting it."

**`Etl::Importers::Generic` and `Etl::Importers::SdwisViols` do not do this.** Both call
`imported_result` with no metadata, so `changed_pwsids` is always `[]`. `SdwisViols` is the importer
that populates `phone_number`, `owner_type`, `years_operating`, and the violations summary fields —
meaning a genuine upstream data change to any of those columns today produces **zero** automatic
tile refresh, selective or otherwise. This is a standing gap, independent of any code change, and
would be fixable by applying the same `changed_pwsids_for` diff-before-upsert pattern to those
importers.

### Full bust + warm (explicit full-refresh fallback, broad scope)

`Etl::PostImportSteps.bust_tile_cache` (`TileCache.delete_all` — every layer, every row) plus
`TileCacheWarmJob` is reserved for cases where the *entire* cache needs to be considered stale at
once — e.g. a full `epa_sabs_geoms` geometry refresh, or a broad cartographic boundary reload.
`Etl::PostImportSteps.refresh_boundary_layers` uses the narrower `bust_cartographic_boundary_tile_cache(layers)`
for the `states`/`counties`/`places` layers specifically when only those change.

None of this is scoped by pwsid — it's an "assume everything in scope is stale, regenerate all of
it" operation, appropriate when the change genuinely is that broad.

## Manual invalidation — required for code changes, not just data changes

Everything above is triggered by *data* changing, signaled through an `ImportResult`. None of it
can detect a *code* change to what `TileGenerator.layer_sql` selects for a layer (e.g. adding a new
column to the `pws` attrs, as in the map-popup detail fields added in mid-2026). That's invisible
to the ETL freshness/diff machinery entirely — no import ran and nothing changed in the database,
so there's nothing for any importer to report.

**Whenever a PR changes which columns/attributes a tile embeds for an existing layer, the cache for
that layer must be busted manually in every environment after deploy** — staging, production, and
preview each hold their own `tile_cache` rows, and none of them clear automatically just because
new code shipped. Existing cached rows keep serving the old shape (missing/blank new fields)
indefinitely on a cache hit; only a cache miss regenerates with current code.

This is also why a "surgical" bust doesn't apply to code changes the way it does to data changes —
a code change potentially affects *every* previously-cached row for that layer, regardless of
whether any given region's underlying data changed at all. There's no diff to scope against; a
full-layer bust is the correct, unavoidable scope.

Use the **Bust Tile Cache** and **Warm Tile Cache** GitHub Actions workflows to do this per
environment — see [RUNBOOK.md](RUNBOOK.md#tile-cache) for the input reference and
[DEPLOYMENTS.md](DEPLOYMENTS.md#bust-tile-cache) for how they work under the hood.

**Local dev:** anyone pulling a branch that changed tile SQL, with an existing populated local DB,
needs to run the same bust manually — a fresh/never-seeded local DB doesn't, since it has no stale
rows to begin with:

```bash
bin/rails runner 'TileCache.where(layer: %w[pws pws_low_poly_v1]).delete_all'
```

## Pre-warming (`TileCacheWarmJob`)

`TileCacheWarmJob` pre-generates tiles for z0–z8 using hardcoded US region bounding boxes
(continental US, AK, HI, PR, Guam+CNMI) — it skips empty ocean/land tiles and covers through
city/district zoom, the level where users first interact with individual water utility polygons.
z9+ tiles are never pre-warmed; they generate on-demand (fast, since each covers a small area).

**It is not surgical.** Its only two scoping parameters are `layers:` (which source layers) and
`max_zoom:` (how deep) — within that scope it always iterates the *entire* precomputed set of
US-bounding-box tile coordinates for each zoom level, and `TileGenerator.generate_tile!` always
regenerates and overwrites, with no check for whether a given tile was actually stale. Contrast
with `TileCacheRefreshJob` above, which operates on a specific, bounded coordinate list derived
from what actually changed. They're two different tools: the warm job is a broad, blind
pre-generation sweep; the refresh job is targeted and data-change-driven.

Warming `pws` at any `max_zoom` ≥ 0 automatically covers `pws_low_poly_v1` too — the job always
starts at z0, and `TileGenerator.cache_layer` remaps `pws` requests below z5 to the low-poly key
internally. There's no separate low-poly value to pass; passing `pws_low_poly_v1` itself as a
`layers:` value warms nothing (it isn't a `TileGenerator.layer_sql` case).

The warm job takes roughly **half an hour at national scale** (~44k systems, z0–z8, all 4 layers).
See `scratch/performance_work.md` for full timing data.

## Queues and environments

`tile_refresh` (selective) and `tile_warm` (full pre-warm) are separate SolidQueue queues from
`etl`. Web services run `SOLID_QUEUE_ROLE=web` and never process any of the three — dedicated worker
services run `SOLID_QUEUE_ROLE=worker` and own all of them. In local development, queued
refresh/warm jobs only run automatically if a SolidQueue worker process is running; the ETL itself
runs as a short-lived rake task (`bin/rails etl:import`) regardless.

### Running a full local warm

After loading the national dataset locally, you can warm the full z0–z8 cache manually — useful for
cold-cache performance testing, or to guarantee every tile reflects current code without spot-checking
region by region:

```bash
bin/rails runner "TileCacheWarmJob.perform_now"
```

This runs in the foreground and **blocks that terminal** (not the app) until every z0–z8 tile is
generated — roughly half an hour at national scale. Progress logs to stdout. It's not required for
day-to-day feature verification — a manual bust plus normal browsing (which regenerates on demand
for whatever you're actively looking at) is enough for that; the full warm is closer to a whole-app
sanity sweep.

## Manual workflows (deployed environments)

| Workflow | Scope | Use it when |
|---|---|---|
| `bust-tile-cache.yml` | Deletes `tile_cache` rows for given layers, one environment at a time | A code change altered what an existing layer's tiles embed |
| `warm-tile-cache.yml` | Regenerates + caches tiles for given layers/zoom range, one environment at a time | After a bust, so the first real visitor doesn't pay the live-generation cost |

Both are `workflow_dispatch`-only, gated by a typed confirmation phrase and whatever approval rules
are configured on the `staging`/`production`/`pr-previews` GitHub Environments. Full input reference
and step-by-step usage: [RUNBOOK.md](RUNBOOK.md#tile-cache) and
[DEPLOYMENTS.md](DEPLOYMENTS.md#bust-tile-cache).

Neither workflow runs automatically on deploy — a deploy that changes tile SQL does not, by itself,
clear anything in staging/production/preview. The manual step is required every time such a change
ships.

## Local troubleshooting: multiple app checkouts sharing one database

If you run more than one checkout of this repo locally (e.g. a git worktree on a different branch)
against the **same** local Postgres database, both servers read and write the same `tile_cache`
table. Whichever server last served a given `(layer, z, x, y)` "wins" for that row — a server
running older code can silently re-poison an already-busted cache the next time it happens to serve
that tile. This produces confusing, seemingly zoom- or location-dependent staleness that has nothing
to do with any application bug. If you hit this, stop the stale server (or point one checkout at a
separate local database) before re-running a bust — otherwise it won't stick.
